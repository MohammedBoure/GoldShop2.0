# GoldShop

نظام إدارة محل مجوهرات يعمل كتطبيق سطح مكتب على Windows مع واجهة ويب
مرافقة للقراءة والمتابعة. يدير المخزون بالوزن أو بالقطعة، نقطة البيع،
الفواتير ودفعات العملاء (`versement`)، الموردين، الخزينة، المصروفات،
الإصلاحات، التقارير والطباعة.

هذه الوثيقة تصف الكود الحالي كما هو في **26 مايو 2026**. للتفاصيل الكاملة
ابدأ من [فهرس الوثائق](docs/README.md).

## ما الذي يدعمه النظام؟

| المجال | الوظائف الموجودة |
| --- | --- |
| المخزون | أصناف موزونة `WEIGHT` وأصناف بالقطعة `PIECE`، باركود، مواقع تخزين، حجز، كلفة ومصنعية وهوامش |
| المبيعات | نقطة بيع، خصم، جلسة صندوق، بيع كامل أو جزئي، تسليم جزئي للوزن، سجل مبيعات |
| العملاء | حساب عميل، ديون، دفعات حرة، دفعات مرتبطة بفاتورة، استعمال رصيد سابق وطباعة مستندات |
| الوثائق | `FACTURE` ورقم بيع، و`BON DE VERSEMENT` برقم `VRS-xxxxxx`، PDF وطباعة حرارية |
| الشركاء | موردون وحرفيون، حسابات `LOCAL/IMPORT`، سندات وزن/أجرة/مبلغ، تسويات ذرية وكشوف قابلة للتصدير |
| المالية | خزائن وصناديق، تحويلات، معادن فعلية، مصروفات، تدقيق وإغلاق الجلسات |
| الخدمات | إصلاحات، تسليم واستلام من الحرفي، ودائع ذهب عينية للعملاء |
| الإدارة | مستخدمون وصلاحيات، بيانات مرجعية، تخصيص الواجهة والطابعات والنسخ الاحتياطي |
| الويب | صفحات متابعة وAPI محمي بكلمة مرور، العربية/الفرنسية، تحديث سعر الذهب المصرح به |

## التقنيات

- Python و`PySide6` لواجهة سطح المكتب.
- Flask لصفحات الويب وREST API المدمج.
- MySQL مع `mysql-connector-python` وSQLAlchemy.
- `pandas` لتصدير واستعادة بيانات النسخ الاحتياطي.
- `pywin32` والطباعة عبر Qt لطابعات الفواتير والملصقات.

## تشغيل سريع

### المتطلبات

- Windows، لأن مسارات الطباعة الحالية تعتمد على واجهات Windows.
- Python 3 حديث؛ الاختبارات الحالية شغلت باستخدام Python 3.12.
- خادم MySQL قابل للوصول.

### التثبيت

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

أنشئ ملف `.env` في جذر التطبيق أو بجانب الملف التنفيذي المجمع:

```dotenv
DB_HOST=localhost
DB_PORT=3306
DB_USER=goldshop_app
DB_PASSWORD=change-this-password
DB_NAME=JewelleryDB
READ_API_ALLOWED_ORIGINS=http://localhost:8000
GOLDSHOP_ACTIVATION_SECRET=replace-with-the-same-secret-used-by-Jewelry-in-ActivateurRMS
```

يفضل إنشاء مستخدم MySQL محدود مخصص للتطبيق بدلا من تشغيله بحساب `root`.

### تفعيل البرنامج

يدعم GoldShop تجربة محلية لمدة 7 أيام لكل جهاز. بعد انتهاء الفترة التجريبية
يعرض البرنامج `Code demande` ويجب إدخاله في برنامج `ActivateurRMS` مع اختيار
`Jewelry`، ثم إدخال مفتاح التفعيل الناتج في GoldShop.

يجب أن تكون قيمة `GOLDSHOP_ACTIVATION_SECRET` في ملف `.env` هي نفس سر
`KEYGEN_TOKEN_JEWELRY_SECRET` المستخدم في مولد المفاتيح. لا تضع هذا السر في Git.

### تشغيل سطح المكتب

```powershell
python main.py
```

عند أول تهيئة ينشئ النظام قاعدة البيانات والجداول والبيانات المرجعية
الأساسية، كما ينشئ المستخدم الأولي:

```text
username: admin
password: admin123
```

يجب تغيير كلمة المرور فور أول دخول، وعدم استعمال بيانات الاعتماد الأولية في
متجر فعلي.

بعد نجاح تسجيل الدخول يبدأ خادم Flask في الخلفية على:

```text
http://localhost:8000
```

واجهة API لا تقبل القراءة قبل ضبط كلمة مرور الويب من
`Parametres > Configuration > General > Mot de passe API Web`.

```text
GoldShop/

|-- database/               # الاتصال، المخطط، managers وقواعد الأعمال
|-- ui/                     # النوافذ والحوارات والطباعة وأدوات Qt
|-- templates/              # صفحات Flask
|-- translations/           # ترجمة الويب ar/fr
|-- scripts/                # البناء وأدوات الصيانة
|-- test/                   # اختبارات unittest المنظمة حسب المجال
|-- docs/                   # وثائق الاستخدام والإدارة والتطوير
```

## مسار العمل المعتاد

1. اضبط البيانات الأساسية: أنواع المعدن، الأصناف، المواقع، العملات،
   الصناديق والخزائن، ومعلومات المتجر.
2. أضف المخزون مع الباركود ونوع البيع (`WEIGHT` أو `PIECE`) والكلفة والسعر.
3. افتح جلسة صندوق من نقطة البيع.
4. نفذ البيع نقدا أو بعربون/دفعة جزئية، أو استعمل `versement` متاحا للعميل.
5. اطبع `Facture` أو `Bon de Versement` بحسب المستند المطلوب.
6. راقب الديون والدفعات من `Versements & Dettes`.
7. أغلق الجلسة بعد العد والتدقيق، وفعل النسخ الاحتياطي التلقائي قبل
   الاستعمال الفعلي.

## الفاتورة والـ Versement باختصار

- رقم الفاتورة هو رقم سجل البيع `Sales.id` ويعرض عادة بصيغة مثل `00321`.
- رقم `versement` يأتي من سجل العملية `VersementOperations.id` ويعرض مثل
  `VRS-000324`.
- `VRS` تسلسل مستقل من حيث الجدول والهوية عن رقم الفاتورة، لكنه تسلسل عام
  لعمليات الـ versement في قاعدة البيانات ولا يبدأ من `1` لكل عميل.
- الدفعة على منتج تبقى مرتبطة برقم الفاتورة، لذلك يظهر في مستندها أيضا
  `Facture N°`.
- الدفعة الحرة غير المرتبطة ببيع تظهر كـ `Versement libre` ويمكن استهلاك
  رصيدها لاحقا.

التفاصيل التشغيلية والقيود الحالية موثقة في
[المبيعات والدفعات](docs/sales-and-versements.md).

## دفتر الموردين وملف Excel القديم

يدعم دفتر الموردين سندات `Poids / Afaçon / Montant` والحسابات الفرعية
والتسويات والتقارير، مع مسار staging/mapping/reconciliation لنقل
`Credit Fournisseurs 2026.xls`. لا ينشر التحليل وحده بيانات تشغيلية، ولا
تدخل أوراق `TPE` أو الكسر أو الحسابات العامة إلى الموردين تلقائيا.

لا يجوز إيقاف تحرير الملف الحقيقي قبل backup مجرب ومطابقة موقعة لكل
مورد وحساب. اتبع
[Runbook نقل دفتر الموردين والقطع عن Excel](docs/supplier-legacy-import-cutover-runbook.md).

## تنبيهات مهمة قبل بيع البرنامج أو نشره

- غيّر المستخدم وكلمة المرور الافتراضيين، واستخدم مستخدم MySQL مخصصا.
- راجع ترقية تخزين كلمات مرور مستخدمي سطح المكتب قبل الإنتاج؛ التنفيذ
  الحالي يعتمد hash `SHA-256` مباشر.
- لا تعرض المنفذ `8000` للإنترنت دون HTTPS، حماية شبكة، وكلمة مرور API قوية.
- `config.json` قد يحتوي token لـ DuckDNS وإعدادات حساسة؛ لا تشاركه علنا.
- فعّل النسخ الاحتياطي واختبر الاستعادة على نسخة منفصلة قبل الاعتماد عليه.
- أدوات `scripts/maintenance/` وبعض عمليات الاستعادة يمكن أن تحذف بيانات؛
  لا تستخدمها على قاعدة إنتاج دون نسخة احتياطية مؤكدة.
- راجع [جاهزية الإنتاج](docs/production-readiness.md)؛ وجود الوظائف لا يعني
  تلقائيا أن النظام مستوف لكل متطلبات الأمان أو الالتزام القانوني لمتجرك.
